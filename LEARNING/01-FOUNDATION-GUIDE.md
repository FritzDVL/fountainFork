## What is the Auth Server? (The Big Picture)

Think of your app as a nightclub. There are three players:

```
YOUR APP (the nightclub)     ←→     LENS API (the city government)
                                         ↕
                              YOUR AUTH SERVER (the bouncer)
```

When a user tries to log in to your app:

1. User clicks "Connect Wallet" on your app
2. Your app asks the Lens API: "Hey, this person wants to log in"
3. Lens API calls YOUR auth server: "Should I let this person in?"
4. Your auth server responds: "Yes, let them in" (or "No, block them")
5. Lens API gives the user a session token
6. User is now logged in

**Without an auth server:** Lens has no way to ask you "is this person
allowed?" — so by default, ANYONE can log in. That might be fine for
testing, but for a real app you want control.

**The auth server is just a tiny web server** (like 50 lines of code)
that answers one question: "Should this user be allowed in?"

---

## Do I need the auth server running at the same time?

**Yes, but not on your computer.** Here's the key insight:

- Your app (Next.js) runs on `localhost:3000` during development
- The auth server needs to be **publicly accessible on the internet**
  because the Lens API (which runs on Lens's servers) needs to call it

The Lens API is not on your computer. It's on the internet. So when it
wants to ask "should this user be allowed?", it needs to reach your auth
server via a public URL like `https://auth.yourdomain.com/authorize`.

**During development**, you have two options:
1. Deploy the auth server to a free service (Railway, Render, Fly.io)
2. Use a tunnel (ngrok) to temporarily expose your local server

**For production**, you deploy it once and forget about it. It just sits
there answering yes/no questions.

## What are "Auth Keys"?

When you run `bun src/keygen.ts` in the auth server repo, it generates
three things. Think of them like this:

### 1. SIGNER_PRIVATE_KEY (the stamp)

This is a secret key that your auth server uses to "stamp" every action
a user takes. When someone posts an article through your app, this stamp
proves it came from YOUR app, not a fake copy.

It's like a notary stamp — only you have it, and it proves authenticity.

### 2. SIGNER_ADDRESS (the stamp's ID number)

This is the public identifier of your stamp. You register this with Lens
so they know: "When I see this stamp on an operation, it's legit — it
came from Society Protocol's app."

### 3. AUTH_API_SECRET (the password between you and Lens)

This is a shared secret between your auth server and the Lens API. When
Lens calls your auth server, it includes this secret in the request
header. Your auth server checks: "Is this really Lens calling me, or
some random person?" If the secret doesn't match, it rejects the request.

### Why are there TWO private keys?

This confuses everyone. Here's the difference:

```
PRIVATE_KEY (your Builder wallet)
  → The wallet that OWNS the Lens App
  → Used ONCE to register things with Lens (like a CEO signing contracts)
  → You already have this — it's in your .env.local

SIGNER_PRIVATE_KEY (generated by keygen)
  → A NEW key that SIGNS operations
  → Used by the auth server every time someone does something
  → Like an employee badge that stamps every document
```

They serve completely different purposes. Never use one for the other.

## What does the Auth Server actually DO? (In detail)

The auth server has ONE job. It's an Express.js web server with a single
endpoint: `POST /authorize`.

Here's what happens in real life:

```
1. User clicks "Login" on your forum
2. Your app → Lens API: "User 0xABC wants to log in to app 0x637E"
3. Lens API → Your auth server: POST /authorize
   {
     "account": "0xABC...",     ← the user trying to log in
     "signedBy": "0xABC..."     ← the wallet that signed the request
   }
4. Your auth server thinks about it...
   (Is this user banned? Are they a member? Do I care?)
   ...and responds:
   {
     "allowed": true,           ← yes, let them in
     "sponsored": true          ← yes, I'll pay their gas fees
   }
5. Lens API → User: "Here's your session token, you're in"
```

That's it. The entire auth server is basically:

```js
app.post('/authorize', (req, res) => {
  // For now, let everyone in
  res.json({ allowed: true, sponsored: true });
});
```

Later, you can add logic like:
- Check if the user is banned
- Check if they're a group member
- Check if they hold a specific token

But for Phase 1, you just let everyone in. Keep it simple.

### The 500ms rule

Lens requires your auth server to respond within 500 milliseconds.
If it takes longer, Lens denies the login. This is why:
- The auth server should be lightweight (no heavy database queries)
- It should be deployed close to Lens's servers (any major cloud works)
- Serverless (Vercel) can have cold start issues — a long-running server
  on Railway or Render is safer

## "Register with Lens" — What is a Builder?

This was the confusing part for you, and honestly it's confusing for
everyone. Let me break it down.

### Lens has 4 types of users:

| Role | Who | What they can do |
|---|---|---|
| Account Owner | A regular user | Post, follow, react — normal social stuff |
| Account Manager | A delegated user | Act on behalf of an Account Owner |
| Onboarding User | A new user | Create an account, that's it |
| **Builder** | A developer (YOU) | Configure apps, create groups, create feeds |

### How do you "become" a Builder?

You don't apply or register anywhere. **Builder is just a login mode.**

When you log in to the Lens API, you choose HOW to log in:

```ts
// Regular user login:
client.login({
  accountOwner: {
    account: "0xMyAccount",
    app: "0xMyApp",
    owner: signer.address,
  }
});

// Builder login (that's you, the developer):
client.login({
  builder: {
    address: signer.address,  // your wallet address
  }
});
```

That's it. You log in as `builder` instead of `accountOwner`. No
application process, no approval. If you own a wallet, you can log in
as a Builder.

### What can a Builder do that a regular user can't?

- Create Lens Apps
- Create Groups and Feeds
- Register authorization endpoints
- Add app signers
- Configure sponsorships
- Manage app settings

### Why wasn't this clear on the Lens Developer Platform?

Because the Developer Dashboard is a GUI that handles Builder auth
behind the scenes. When you use the dashboard, you're already logged
in as a Builder — you just don't see it. The confusion happens when
you try to do the same things via code (scripts), where you need to
explicitly authenticate as a Builder.

### The registration script

The `register-auth-endpoint.ts` script does this:

```
1. Log in to Lens API as Builder (using your wallet private key)
2. Tell Lens: "My app's auth endpoint is at https://auth.mysite.com/authorize"
3. Tell Lens: "Here's the secret to use when calling my endpoint"
4. Tell Lens: "Here's my signer address — trust operations signed by it"
```

This is a ONE-TIME operation. You run it once, and Lens remembers your
configuration forever. You never need to run it again unless you change
your auth server URL or signer key.

## The Complete Flow (Everything Connected)

Here's how ALL the pieces fit together:

```
SETUP (one-time, you do this once):
  1. You have the Fountain fork (your app code) ✓
  2. You create a Supabase project (your database)
  3. You generate auth keys (keygen.ts)
  4. You deploy the auth server (Railway/Render)
  5. You run the registration script (tells Lens about your auth server)

RUNTIME (every time a user interacts):
  User → Your App (Next.js) → Lens API ↔ Your Auth Server
                                  ↓
                            Lens Protocol (onchain)
                                  ↓
                            Grove Storage (content)
```

### What runs where:

| Thing | Where it runs | When |
|---|---|---|
| Your app (Next.js) | localhost:3000 (dev) or Vercel (prod) | Always |
| Auth server (Express) | Railway/Render/Fly.io | Always |
| Supabase (database) | supabase.com (cloud) | Always |
| Lens API | Lens's servers | Always (not yours) |
| Registration script | Your terminal | Once, then never again |
| Keygen script | Your terminal | Once, then never again |

## Common Mistakes to Avoid

1. **Using PRIVATE_KEY where SIGNER_PRIVATE_KEY should go** (or vice versa)
   - PRIVATE_KEY = your wallet (the CEO)
   - SIGNER_PRIVATE_KEY = the generated key (the employee badge)

2. **Auth server not publicly accessible**
   - Lens API can't call `localhost:3004` — it needs a real URL
   - Use Railway, Render, or ngrok for development

3. **Auth server too slow**
   - Must respond in < 500ms or Lens rejects the login
   - Keep it simple — just return `{ allowed: true, sponsored: true }`

4. **Wrong Supabase client**
   - Browser components: `@/lib/db/client`
   - Server/API routes: `@/lib/db/server`
   - Scripts: `@supabase/supabase-js` with service key

5. **Forgetting placeholder env vars**
   - Fountain's env schema requires LISTMONK and IFRAMELY vars
   - Just put `placeholder` — they're not needed for the forum
